Node Names and File System Paths

The last argument to all pipeline(1) commands is a name which identifies the data/process node. All nodes have a unique name in the database which resembles a file system path. For example, a data node associated with a set of RIB files could be named:
"/projectX/maya/rib/deathray"
Its important to note that this name is not a literal file system path. However, the above name is used in combination with several other node properties to construction literal file system paths.

Files and File Sequences

Process nodes can only be associated with a single script file which when executed performs the process. Data nodes, however, are a little more complicated. Some data nodes are associated with a single file, while others are associated with a sequence of files which share a common naming convention. Examples of single data nodes include texture maps, models and scene files.  Sequence data nodes are commonly used for sequences of images in an animation, per-frame input files for renderers (such as RIB files) and data files associated with simulations.  

In order to support this complexity, several options to the register command are provided to which allow the user to specify frame ranges (--range), frame number padding (--padding) and file name suffixes (--suffix).

One important restriction to keep in mind is that the padding, frame range an suffix properties of a data node are fixed at the point of node registration and cannot be modified once a node has been checked-in. The reason behind this rather severe restriction is that the literal file names associated with nodes are commonly referenced inside dependent nodes downstream and any change in file naming convention would cause widespread errors and failures when these dependent nodes are later evaluated. If a change in required in these properties, a new node (with a different name) should be created by cloning the old node instead (--clone).

Working Area Paths

The path most important to users is the location of the associated files within their per-user working area directories. Each user is assigned a directory with the same name as their username under the central working area (see --working option). Under this per-user directory, the structure mirrors that of the repository but only contains versions of files which are actively being used, hence the name "working" area.  

In order to facilitate the sharing of files with internal references to file system paths, the $WORKING environmental variable is defined by pipeline(1) when executing process node scripts and launching editors. Users should always use this environmental variable (instead of its value) in any file system path references to insure that the the paths remain valid when used by another user with a different working area location.

The value of $WORKING is set to the fully qualified filesystem path of the root of the user's working area directory structure. More specifically, $WORKING is set as if the following bash shell script fragment was executed:
> WORKING=`pipeline --working`/$USER
For example, assuming a padding of "4", a suffix of "rib" and a frame range of "1-5x2", the path to the working files associated with the example node above are:
"$WORKING/projectX/maya/rib/deathray.0001.rib"
"$WORKING/projectX/maya/rib/deathray.0003.rib"
"$WORKING/projectX/maya/rib/deathray.0005.rib"
Notice that the above paths contain no mention of the specific revision of "deathray" stored under the per-user working area. Version information is stored in the database instead of cluttering up file names. This way any references to "deathray" stored in other files or scripts will never have to be modified when newer versions of "deathray" are released. All such references should use fully qualified working area paths similar the previous example. This protects users from unexpected changes in the naming of files they use while simplifying the steps required to use new versions.

Working files may be either copies or symbolic links to a specific revision stored in the repository depending on the options used during file check-out. Generally, symbolic links are only used for files which are large and do not need to be locally modified.

Repository Paths

All users working on the same production share a central repository directory structure where all revisions which have been checked-in are stored (see the --repository option). Continuing with the example, the "2.5.1" version of the "deathray" sequence of RIBs would be stored in the repository as:
"<repository>/projectX/maya/rib/deathray/v2.5.1/deathray.0003.rib"
"<repository>/projectX/maya/rib/deathray/v2.5.1/deathray.0003.rib"
"<repository>/projectX/maya/rib/deathray/v2.5.1/deathray.0005.rib"
All files and directories in the repository are read-only and should never be directly modified or referenced by an artist. Instead, the pipeline(1) tool should be used to perform all changes to the contents of the repository. Due to these restrictions, there is no equivalent environmental variable to $WORKING defined for the repository path.

Relative Names

The (name) argument to pipeline(1) commands can always be given as a fully qualified node name (i.e. without the $WORKING prefix or possible file sequence suffixes). As a convenience, when pipeline(1) is executed within a subdirectory of $WORKING, a shorter relative path may be used to refer to nodes. For instance, the following would be a valid way of referring to the example node:
> cd $WORKING/projectX/maya
> pipeline command [options ...] rib/deathray

Editors

For each node, pipeline(1) stores the name of an appropriate editor for viewing and modifying the data files associated with the node.  The editor can be set for a node using the --editor option.  

Because of the complexities of file sequences and the arbitrary nature of command line options for editor programs, pipeline(1) uses Java based plugins for each registered editor to handle the specific calling conventions of the editor executables.  Most useful editor programs for the CG industry are already covered by pipeline(1), but this list can be extended by writing a custom Java plugin for your site.  See the Pipeline Plugins document for a full explanation of this mechanism.

Actions

One of the primary features of pipeline(1) is the ability to regenerate the data file(s) associated with a node through execution of some process on the local machine or render queue. This process is refered to as the regeneration action (or just action) for a node.  Each node has one of the following associated actions:

None

The simplest possible action which does nothing.  This is the default action for nodes (see --noaction).  This action has no parameters.

Script

This action executes a text script using an arbitrary interpreter. This highly flexible action can be used to do just about anything. The only limitation of ScriptAction's is that the script is only passed the most basic information about its associated node and therefore must be hardcoded to do a specific job.  By contrast, most of the other actions are very generic and produce very different results depending on the node with which they are associated. ScriptActions have the following parameters:

Interpreter

This is the name of program which will be used as the interpreter for the ScriptText.  This can be any program that can be used with the "!#" notation as the first line of a shell script.  

Common values include: "bash", "sh", "csh" and "perl".

ScriptText

The literal text of the script to be interpreted by the Interpreter.  

MayaRender
Render a Maya scene file using the Maya renderer.  Generates one or more images depending on the execution mode.

RenderScene

The name of the node who's primary file is the Maya scene to render.  This render scene node must also be one of the dependencies of the node which uses this action.  If there is only one dependency and it is the render scene node, this parameter will default to the dependency if it is left unset.
