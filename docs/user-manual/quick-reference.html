Node State Symbols
Symbols
 Description

Pending:
The Pending state means that the node was created by you but has not yet been checked-in.  At this point, you are the only artist which can seen this node in Pipeline and its associated file(s) exist only in your working area.

Identical:
The Identical state means that the node's working area properties, dependencies and associated files are all identical to those of the latest version of the node in the repository.

Modified:
The Modified state means that one or more of the node's working area properties or associated files have been changed since the node was last checked-out.  Your changes were based on the latest version of the node.
Modified Dependencies:
One or more of the node's dependencies have been modified, but otherwise the node is identical to the latest version in the repository.  Adding a new dependency, removing a previously existing dependency and changing the dependency options are all considered as modifications of the dependencies.  In addition, the Mofied Dependencies state can be cause by one of the dependencies of the current node having a state other than Identical.
Needs Check-Out:
This state means that someone else has checked-in a new version of the node since the last time you checked-out the node. In addition, no changes have been made to your working copy of the node since the node was checked-out.
Conflict:
The Conflict state occurs when both the working copy and repository version of the node have changed since the last check-out.  This can happen when two artists check-out the same version of a node and both modify their working copies.  Once the first artist checks-in their changes, the second artist will see a Conflict state for the node. 

At this point, the two artists will need to talk to each other to resolve how to combine their changes. Nodes cannot be checked-in when they are in the Conflict state. The second artist will need to check-out the new version created by the first artist and then modify it to include any of their changes they want preserved in the combined version.

Checked-In:
This state means that the node has not been checked-out.  In other words, the node exists in the repository but does not exist in your working area.

Check-In... This operation creates a new version of the node and any of its modified dependencies in the repository.  The node must be in a Finished (blue) state before Pipeline will allow the check-in.  Once a node is checked-in, other artists will be able to see the node and check-out a copy of the node into their own working area. The check-in options are:

New Version:
Inc Minor
Increments the most minor version of the node.  For example, if the previous latest version was (v2.4.6) then the new version will be (v2.4.7).

Add Minor
Extends the levels of the version by one sublevel.  For example, if the previous latest version was (v2.4.6) then the new version will be (v2.4.6.1).

Inc Major
Increments the lowest major level of version. For example, if the previous latest version was (v2.4.6) then the new version will be (v2.5).

Check-Out...
 This operation replaces (or creates) the working area copy of the node with a version from the repository.  The check-out options are:

Version:
The version number to check-out.  The "Latest" choice will check out the most current repository version.  Older versions can be checked-out by chooseing a specific version number.

Local Mods:
Abort Check-Out
Cancels the check-out if any of the nodes being checked-out have been modified in the working area.  This prevents the check-out operation from unintententionally overwriting changes you haven't check-in.  If the check-out is aborted, then none of the nodes will have been modified.

Force Overwrite
Performs the check-out regarless of the state of the nodes.  Any changes which have not been checked-in will be lost!

File/Queue State Colors
Colors
 Description

Missing:
One or more of the files associated with the node do not exist in the working area.  If this is a leaf node, then the file will need to be recreated or an older version check-out from the repository.  The more common case is that this is a branch node which has a regeneration action that can be run to create the missing file(s).
Stale:
The node's properties have been modified since the time that the files associated with the node were saved.  A node can also become Stale if one if its dependencies or the files associated with the dependencies are modified since the last time the the target node's files where changed.  In other words, changes upstream of this node require that this node be regenerated to reflect the changes.

Queued:
One or more jobs have been submitted which will regenerate files associated with the node, but none of these jobs has started running yet.
Running:
One or more of the jobs is currently running.
Error:
At least one of the submitted jobs failed with errors.
Finished:
All jobs submitted for this node have completed successfully and all associated files are up-to-date.  Nodes must be in a Finished state before they can be checked-in.
